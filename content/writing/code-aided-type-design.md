從手動調整到腳本自動化，這條路走了大約一年。回頭看，每一步都是自然而然的——不是為了寫程式而寫程式，而是因為手動做太痛苦了。

## 起點：重複的痛苦

設計漢字字型時，有些工作是純粹重複的。比如說，你需要把某個部件的修改同步到所有包含它的字符。在 Glyphs 中，如果你用了 component（部件參照），這個問題可以自動解決。但如果部件已經被分解成路徑（decompose），你就得一個一個手動改。

一開始我真的是一個一個改。改到第 50 個字的時候，我開始想：「一定有更好的方法。」

## 第一個腳本

我的第一個 Glyphs 腳本只有不到 20 行。它做的事情很簡單：遍歷所有選中的字符，把特定位置的節點移動固定的距離。

```python
for layer in Font.selectedLayers:
    for path in layer.paths:
        for node in path.nodes:
            if node.y > 700:
                node.y += 10
```

這段程式碼省了我大概兩個小時的手動工作。那個瞬間，我理解了為什麼程式設計師會說「懶惰是美德」。

## 從腳本到外掛

腳本解決了「批次處理」的問題，但有些需求更複雜。比如，我想在編輯字符的時候，即時看到它在不同上下文中的效果。這就需要一個能持續運行、能回應使用者操作的工具——也就是外掛。

NineBoxView 就是這樣誕生的。它的核心功能是把當前編輯的字符放在一個 3×3 的九宮格中，周圍用其他字符填充，讓你能即時看到字符在文字排列中的效果。

開發外掛比寫腳本困難得多。你需要理解 Glyphs 的 callback 機制、Cocoa 的 UI 框架、Python 和 Objective-C 的橋接。但每解決一個技術問題，都會帶來設計工作流程的顯著改善。

## 工具改變思維

有了自動化工具之後，我的設計方式也改變了。以前我會避免做大範圍的調整，因為「改起來太麻煩了」。現在，我可以大膽地嘗試不同的方案，因為我知道撤銷和重做的成本很低。

這就是工具的力量：它不只是讓你做得更快，還讓你願意去嘗試更多可能性。

## 給想學程式的設計師

如果你是字型設計師，想開始用程式碼輔助工作，我的建議是：

1. 從解決一個具體的痛點開始，不要為了學而學
2. Python 是最好的起點，Glyphs 原生支援
3. 先寫腳本，等到有需要再學外掛開發
4. 不需要成為「程式設計師」，只需要成為「會寫程式的設計師」

當重複的工作交給程式，設計師就能專注在真正需要判斷力的地方。
