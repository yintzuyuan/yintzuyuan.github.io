2024 年初，我發布了 NineBoxView——一個 Glyphs 字型編輯器的九宮格預覽外掛。從一個簡單的腳本發展到完整的外掛產品，這個過程中有很多技術選擇和設計考量值得分享。

## 為什麼是外掛而不是腳本

Glyphs 的腳本（Script）和外掛（Plugin）有本質的區別：

- **腳本**是一次性執行的程式，適合批次操作
- **外掛**是持續運行的模組，可以回應使用者的操作

NineBoxView 需要在使用者編輯字符時即時更新預覽，這只有外掛才能做到。

## 技術架構

Glyphs 外掛基於 Python，但底層使用 Cocoa 框架（透過 PyObjC 橋接）。這意味著你需要同時理解兩個世界：

```python
# Python 語法，但操作的是 Cocoa 物件
class NineBoxView(GeneralPlugin):
    @objc.python_method
    def start(self):
        # 初始化 Cocoa UI
        self.view = NSView.alloc().init()
```

最大的挑戰是 callback 機制。Glyphs 提供了多個回呼時機——字符切換、圖層修改、視窗縮放——你需要在對的時機更新 UI，同時避免不必要的重繪導致效能問題。

## 設計決策

### 九宮格佈局

為什麼是 3×3 而不是其他佈局？因為這是預覽上下文效果的最小有效尺寸。中間放編輯中的字符，周圍 8 格放上下文字符，剛好可以看到：

- 水平排列的間距效果
- 垂直排列的灰度一致性
- 對角線方向的視覺平衡

### 效能優化

即時預覽意味著每次滑鼠移動都可能觸發重繪。為了保持流暢：

- 使用 `setNeedsDisplay_` 而非直接 `display`，讓系統合併重繪請求
- 快取已渲染的字符圖像，只在字符實際改變時重新渲染
- 限制更新頻率，避免 UI 執行緒阻塞

## 從免費到付費

NineBoxView 最初是免費的開源專案。當功能逐漸豐富（自訂上下文、深色模式、進階設定），我決定推出 Pro 版本。

這個決定背後的考量是：持續維護和開發需要時間投入，而付費模式可以讓這件事可持續。免費版本繼續存在，提供核心功能；Pro 版本提供進階功能和優先支援。

## 學到的事

1. **從使用者的痛點出發**——NineBoxView 解決的是我自己每天遇到的問題
2. **先做最小可用版本**——第一版只有基本的九宮格預覽，沒有任何設定選項
3. **效能是功能**——一個卡頓的即時預覽工具，不如沒有
4. **文件和社群很重要**——好的工具加上差的文件，等於半個好工具
